# Architecture

How this dotfiles repo is laid out and why. Target audience: future me,
and anyone forking it.

## Repository layout

Top-level directories are either **stow packages** (symlinked into `$HOME`
on `make install`) or **assets** (consumed by the install tooling, not
stowed). The Makefile and the installer both consult the same ignore
list:

```text
ai/         stow → ~/.config/{claude,codex,opencode,nono,…} + ~/.local/bin/ai-*
local/      stow → ~/.local/bin/ (dots, theme, fzf-preview, vm, knvim, …)
nvim/       stow → ~/.config/{lazyvim,kickstart}
shell/      stow → ~/.config/fish/
ssh/        stow → ~/.ssh/
term/       stow → ~/.config/ghostty/
tmux/       stow → ~/.config/tmux/
tools/      stow → ~/.config/{bat,git,sesh,…}
vcs/        stow → ~/.config/jj/
zed/        stow → ~/.config/zed/
docs/       asset (screenshots, this file)            ← not stowed
```

`STOW_IGNORE_RE` in the Makefile and a matching `case` in `install`
keep the ignore list in sync. Add a new asset directory in both places.

## The `dots` command surface

`local/.local/bin/dots` is a thin dispatcher to sibling `dots-*`
scripts. Adding a new subcommand means dropping a `dots-foo` script
into `local/.local/bin/` and adding a `case` in `dots`.

```text
dots doctor    → dots-doctor      health check (binaries, symlinks, configs)
dots update    → dots-update      pull, re-stow, refresh plugins
dots versions  → dots-versions    print tool versions
dots keys      → dots-keys        keybinding reference (see @key below)
dots theme     → theme            theme switcher (see Theme pipeline below)
```

## `@key` annotation convention

Keybindings are documented inline in tmux and fish config files using
a single comment format:

```text
# @key tool :: KEY :: human-readable description
```

`dots-keys` greps for this marker across `tmux/` and `shell/`, sorts by
tool, and renders a table. Adding a binding is one line; the reference
stays in sync because the source of truth is the binding itself.

## Theme pipeline

The `theme` script (`local/.local/bin/theme`) is the single entry point
for switching colors. Given a Ghostty theme name, it:

1. Parses Ghostty's theme file to extract the 16 ANSI palette colors,
   background, foreground, and cursor.
2. Rewrites `term/.config/ghostty/config` so new windows persist.
3. Pushes the palette to the running terminal via `OSC 4/10/11/12`,
   broadcasting to every tmux client tty if `$TMUX` is set.
4. Generates a delta config (`tools/.config/git/delta.gitconfig`) with
   blended hex colors derived from the palette.
5. Tells running nvim instances to call `require('theme').apply()`,
   which maps the Ghostty theme name to a colorscheme plugin.

bat and eza are deliberately not regenerated: bat is pinned to
`BAT_THEME=ansi` (in `shell/.config/fish/conf.d/06-theme.fish`) and
eza's defaults use ANSI color numbers. Both ride the OSC palette change
for free.

`theme auto` reads system appearance (`AppleInterfaceStyle` on Darwin,
`gsettings color-scheme` on Linux) and applies a light/dark default.
Run it on demand or wire it into your own scheduler — the repo
deliberately does not install a system-level watcher.

## AI agent rule rendering

Three coding agents (Claude Code, Codex, OpenCode) all want their own
rules file in their own location, but most rules are shared. Source of
truth lives in `ai/.config/ai-shared/`:

```text
AGENTS.base.md           shared rules (sandbox notes, rg > grep, …)
overlays/claude.append.md Claude-specific additions
```

`ai/.local/bin/ai-agents-render` concatenates the base with each
overlay (Codex and OpenCode have no overlay), substitutes
`{{SANDBOX_PATHS}}`, and writes:

```text
ai/.claude/CLAUDE.md
ai/.codex/AGENTS.md
ai/.config/opencode/AGENTS.md
```

Each generated file starts with a `Generated by …` header so nobody
edits them by hand. `make fmt` depends on `make ai-render`, so any
drift between source and rendered files surfaces as a diff in the
formatter step.

## Sandbox profiles

All three AI agents launch through `nono`, a capability-based sandbox
(Seatbelt on macOS, Landlock on Linux). Profiles live in
`ai/.config/nono/profiles/{claude,codex,opencode}.json` and grant only
the working directory plus each tool's config and cache directories.
Launch via `sb claude` / `sb codex` / `sb opencode`.

## Local overrides

Every tool that supports it sources a local file outside the repo so
machine-specific tweaks don't pollute git history. See the "Local
overrides" table in `README.md`.

## Formatters and linters

`make fmt` runs stylua, shfmt, fish_indent, prettier, and taplo over
the file globs collected at the top of the Makefile. `make lint` runs
shellcheck, a headless `nvim +Lazy` plugin check, and markdownlint-cli2.
The fish file glob includes `local/.local/bin/dots-*` and `vm` even
though those scripts have no extension, because they are fish.

## Kickstart fallback

`nvim/.config/kickstart/` is a self-contained nvim config kept
alongside the LazyVim setup as an emergency fallback: if LazyVim
breaks (bad lockfile, plugin author force-push, network down during
plugin sync), `NVIM_APPNAME=kickstart nvim` boots a minimal-but-usable
editor in seconds. It is intentionally not removed.
